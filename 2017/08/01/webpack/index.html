  • packsge.json 字段：

    packsge.json 字段全解

    • name : 这个名字最终会是URL的一部分，命令行的参数，目录名，所以不能以点号或下划线开头；这个名字可能在require()方法中被调用，所以应该尽可能短
    • version ： 必填字段，版本号
    • description ： 可选字段，必须是字符串。npm search的时候会用到。
    • main ： 可选字段。这个字段的值是你程序主入口模块的ID。如果其他用户需要你的包，当用户调用require()方法时，返回的就是这个模块的导出（exports）
    • scripts： 字段是一个对象。它的每一个属性，对应一段脚本。命令行下使用npm run命令，就可以执行这段脚本。
    • author ：
    • license ：
    • devDependencies ： 可选字段。如果只需要下载使用某些模块，而不下载这些模块的测试和文档框架，放在这个下面比较不错。
    • dependencies ： 可选字段，指示当前包所依赖的其他包。

  • 开发中 Server(DevServer)
    • 描述影响 webpack-dev-server(简写为：dev-server) 行为的选项。
      • contentBase : 告诉服务器从哪里提供内容。只有在你想要提供静态文件时才需要。
      • compress: true : 一切服务都启用gzip 压缩：
      • inline ： true （默认）只用在命令行工具(CLI)
      • historyApiFallback: true,当使用HTML5 History API，任意的 404 响应可以提供为 index.html 页面。
      • noInfo ： true 启用 noInfo 后，诸如「启动时和每次保存之后，那些显示的 webpack 包(bundle)信息」的消息将被隐藏。错误和警告仍然会显示。
  • 加载器loader===>
    提供一些加载器
    • loader 是对应用程序中资源文件进行转换。它们是（运行在 Node.js 中的）函数，可以将资源文件作为参数的来源，然后返回新的资源文件。
    • 例如：使用 loader 告诉 webpack 加载 CSS 文件，或者将 TypeScript 转为 JavaScript。首先，安装相对应的 loader
+ 应用程序中，有三种方式使用 loader：
    * 通过配置
    * 在 require 语句中显示使用
    * 通过 CLI   （命令行）

+ 处理文件扩展名（这个属性和热重载是冲突的，有热重载就不要写）
  • 插件Plugins===>
    第三方插件包===>
    插件包
    • 插件是 wepback 的支柱功能。在你使用 webpack 配置时，webpack 自身也构建于同样的插件系统上！
    • 插件目的在于解决 loader 无法实现的其他事。
    • 由于 plugin 可以携带参数/选项，你必须在 wepback 配置中，向 plugins 属性传入 new 实例。
    • 自带插件可直接使用
    • 插件通过 npm 下载，通过 require 引用之后可使用
